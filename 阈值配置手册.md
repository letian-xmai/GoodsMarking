# 商品标注阈值配置手册

更新时间：2026-06-25

本文档用于查询当前商品标注流程中的阈值配置，并说明每个阈值的作用、调整位置和调整影响。当前项目里的阈值还没有集中到统一配置文件，实际生效值分散在 `image_workflow/*.py` 和 `模型训练数据/model/*.json` 中。

## 1. 当前正式流程口径

正式商品标注以 `image_workflow/formal_workflow.py` 为主流程：

1. 从 Excel 按 `outward_code` 分组。
2. 下载完整组内原始图片。
3. 使用 `模型训练数据/model/annotation_model.json` 给图片打分。
4. 使用质量规则、目标一致性规则、手机检测规则生成硬排除标记。
5. 按模型分数从高到低排序。
6. 去除硬排除和近重复图片。
7. 复制到 `商品标注结果/{outward_code}/最终结果/`，最多 40 张。

注意：模型阈值 `0.802` 当前主要用于中间目录 `模型选中/模型排除`，最终结果不是单纯按 `score >= 0.802` 截断。最终结果的关键条件是：无硬排除标记、非 `standard` 参考图、非近重复、按模型分数优先取满目标数量。

## 3. 正式流程全局阈值

| 配置项 | 当前值 | 位置 | 说明 | 调整影响 |
|---|---:|---|---|---|
| `target_count` | `40` | `formal_workflow.py`、`cli.py`、`verification.py` | 每个商品最终结果目标张数 | 调大更容易引入低质图，调小结果更保守 |
| `download_workers` | `4` | `formal_workflow.py`、`downloader.py`、`cli.py` | 单商品下载并发 | 调大下载更快，但请求压力更高 |
| `group_workers` | `3` | `cli.py` | 多商品并发数 | 调大批量更快，但网络和磁盘压力更高 |
| `retries` | `2` | `downloader.py` | 下载失败重试次数，实际最多 3 次请求 | 调大可减少偶发失败，但整体耗时增加 |
| `timeout` | `30s` | `downloader.py`、`training_set.py` | 单 URL 下载超时 | 调大适合慢网络，调小更快跳过坏链接 |
| `dedupe_hamming` | `<=4` | `formal_workflow.py`、`selection.py` | 感知哈希近重复阈值 | 调小保留更多相似帧，调大去重更强 |
| 最终结果复制 | 保持源图文件名和格式 | `formal_workflow.py`、`selection.py` | 最终结果直接复制源图，文件名与 URL 原图一致 | 不再通过文件名写入视角、排序或模型信息 |

## 3.1 下载前整组跳过规则

| 配置项 | 当前值 | 位置 | 说明 | 输出结果 |
|---|---|---|---|---|
| 全 standard 组跳过 | 同一 `outward_code` 下所有 `image_url.lower()` 都包含 `standard` | `formal_workflow.py` | 认为该商品组只有白底/建模参考图，没有可标注实拍图 | 不下载、不跑模型、不生成 `最终结果/` |
| 跳过状态 | `skipped_all_standard` | `formal_workflow.py` | 写入 `selection_report.json` 和 `qa_summary.txt` | `selected_count=0`，`shortfall=40` |
| 逐行结果 | `最终结果是否包含该图片=否` | `formal_workflow.py` | 写入 `selection_report.json` 的 `image_decisions` 和 `image_decisions.csv` | 每个 URL 标记为 `跳过：整组下载链接均为standard参考图` |

注意：该规则只在整组所有下载链接都包含 `standard` 时触发。如果同一商品既有 `standard` 链接又有非 `standard` 链接，则不跳过整组，继续正常下载和筛选；其中 `standard` 图仍只作为参考图，不进入最终结果。

## 4. 正式流程硬排除规则

硬排除规则会阻止图片进入 `最终结果/`。

| 标记 | 当前阈值 | 位置 | 业务含义 | 调整建议 |
|---|---:|---|---|---|
| `is_white_background` | `white_ratio > 0.82` 且 `subject_ratio < 0.35` | `quality.py` | 白底图、建模图、商品孤立图 | 若误排实拍浅背景图，可提高 `0.82` 或放宽 `0.35` |
| `is_low_file_size` | `< 10000 bytes` | `quality.py` | 图片文件过小，通常细节不足 | 用户已明确要求至少 10KB，暂不建议调低 |
| `is_underexposed` | `brightness < 70` 且 `white_ratio < 0.03` | `quality.py` | 画面过暗，难以辨认商品 | 若暗色包装被误排，可降低亮度阈值 |
| `has_multiple_products` | `second_component_ratio > 0.10` 且非边缘裁切 | `quality.py` | 疑似多个商品或多个大前景块 | 对复杂背景敏感，误排时优先看 `model_scores.csv` |
| `is_low_boundary_contrast` | `foreground_contrast < 55` | `quality.py` | 商品和背景边界不清 | 用户多次反馈边界不清，暂不建议放宽 |
| `is_incomplete_product` | `subject_ratio > 0.78` 且 `crop_score > 1.40` 且 `foreground_contrast < 75` | `quality.py` | 商品过近、裁切、不完整 | 若近距离标签清晰图被误排，可提高 `subject_ratio` 或 `crop_score` |
| `has_repeated_product_parts` | 边缘裁切 且组件数 `>=2` 且第二组件占比 `>0.025` 且对比度 `<85` | `quality.py` | 疑似重复商品局部、连续帧或多局部 | 主要防止重复局部入选，调松会增加重复风险 |
| `has_phone_like_object` | 见手机检测章节 | `phone_detection.py` | 图片包含手机 | 用户已明确要求去掉包含手机的图片 |
| `has_other_product_color` | 见目标一致性章节 | `target_consistency.py` | 大面积非目标商品颜色 | 防止混入其他商品 |
| `has_multiple_target_instances` | 见目标一致性章节 | `target_consistency.py` | 同一画面多个目标商品 | 用户明确要求只有一个目标商品 |
| `is_low_target_occupancy` | 见目标一致性章节 | `target_consistency.py` | 目标商品占比过低 | 用于处理人物、背景占比过高的问题 |

## 5. 软标记和记录规则

软标记不会直接阻止进入最终结果，但会进入 `model_scores.csv`，并通过模型分数或质量分影响排序。

| 标记 | 当前阈值 | 位置 | 当前作用 |
|---|---:|---|---|
| `is_blurry` | `edge_score < 2.0` | `quality.py` | 记录为软标记，质量分扣 40 |
| `is_tiny_subject` | `subject_ratio < 0.04` | `quality.py` | 正式流程只记录，旧规则流程硬排除 |
| `is_edge_cropped` | `min(width,height) < 120` 或 `crop_score > 1.45` | `quality.py` | 正式流程只记录，质量分扣 180 |
| `is_low_resolution` | `min(width,height) < 180` 且 `max(width,height) < 260` | `quality.py` | 只记录和扣分，不作为正式硬排除 |

调整建议：

- 如果最终结果仍出现模糊图，优先考虑把 `is_blurry` 加入正式硬排除，或把 `edge_score < 2.0` 提高到 `2.5`。
- 如果最终结果仍出现商品占比过低图，优先增强 `is_low_target_occupancy`，不要只依赖 `is_tiny_subject`。
- `is_low_resolution` 当前没有硬排除，是为了避免误杀人工认为可用的小图；若要更严格，可加入硬排除列表。

## 6. 图片质量计算阈值

| 指标 | 当前值 | 位置 | 说明 |
|---|---:|---|---|
| 分析缩放尺寸 | `128x128` | `quality.py` | 大部分质量指标在缩放图上计算 |
| 白色像素 | RGB 三通道都 `>245` | `quality.py` | 用于 `white_ratio` |
| 前景距离阈值 | `distance > 28` | `quality.py` | 像素与边框背景中位色的颜色距离 |
| 大前景组件面积 | `>=120` 像素 | `quality.py` | 用于多组件和多商品判断 |
| 边缘阈值 | `max(12, edge_percentile_85)` | `quality.py` | 用于边缘裁切评分 |
| 边缘边框宽度 | `12px` | `quality.py` | 计算边缘是否集中在四周 |
| 感知哈希尺寸 | `8x8` | `quality.py` | 输出 64 位平均哈希 |

质量分公式：

```text
quality_score = edge_score * 8 + subject_ratio * 100 - white_ratio * 90
```

质量分扣分：

| 条件 | 扣分 |
|---|---:|
| 白底 | `-200` |
| 模糊 | `-40` |
| 主体过小 | `-60` |
| 边缘裁切 | `-180` |
| 文件小于 10KB | `-120` |
| 低分辨率 | `-160` |
| 过暗 | `-180` |
| 多商品 | `-180` |
| 边界低对比 | `-100` |
| 主体不完整 | `-180` |
| 重复商品局部 | `-180` |

## 7. 目标商品一致性阈值

目标商品一致性基于 `manifest.csv` 中 `source=standard` 的白底建模图建立颜色参考。文件名只用于 URL 原名追溯，不再作为 `standard` 参考图判断依据。

| 配置项 | 当前值 | 位置 | 说明 |
|---|---:|---|---|
| HSV 分析尺寸 | `160x160` | `target_consistency.py` | 目标颜色分析尺寸 |
| standard 有效色相下限 | `80` | `target_consistency.py` | 少于该数量则跳过颜色一致性 |
| 有效颜色条件 | saturation `>45` 且 value `>35` | `target_consistency.py` | 过滤灰白和暗部 |
| 排除白背景条件 | 非 saturation `<30` 且 value `>235` | `target_consistency.py` | 建 reference 时忽略白背景 |
| 允许色相阈值 | `max(12, total * 0.006)` | `target_consistency.py` | 目标商品可接受颜色 |
| 主色阈值 | `max(allowed_threshold, max_count * 0.45)` | `target_consistency.py` | 目标商品主色 |
| 允许色相扩展 | `±10` hue bins | `target_consistency.py` | 放宽目标颜色范围 |
| 主色相扩展 | `±8` hue bins | `target_consistency.py` | 放宽主色范围 |
| 组件最小面积 | `>=120` 像素 | `target_consistency.py` | 统计目标颜色组件 |

目标一致性硬排除：

| 标记 | 当前阈值 | 说明 |
|---|---:|---|
| `has_other_product_color` | `other_ratio > 0.55` 且 `other_component > 0.45` | 大面积颜色不属于目标商品 |
| `has_multiple_target_instances` | 主色组件数 `>=2` 且第二主色组件占比 `>0.075` | 同画面多个目标商品 |
| `is_low_target_occupancy` | 目标主色最大组件 `<0.10` 且次要组件 `>0.35` | 目标商品占比过低，人物或背景占主导 |

调整建议：

- 如果还出现商品占比过低：提高 `target_primary_largest_ratio` 下限，比如从 `0.10` 到 `0.12` 或 `0.15`。
- 如果误杀目标商品颜色较少的包装：降低 `target_secondary_component_ratio > 0.35` 或降低主色占比要求。
- 如果混入其他商品颜色：降低 `other_ratio > 0.55` 或 `other_component > 0.45`，会更严格。

## 13. 常见调整场景

### 13.1 最终结果仍有模糊图片

优先调整：

1. 把 `is_blurry` 从软标记加入 `formal_workflow.py` 的 `HARD_FLAGS`。
2. 或将 `quality.py` 中 `edge_score < 2.0` 提高到 `2.5`。

风险：会减少最终结果数量，更多商品进入 `shortfall`。

### 13.2 最终结果中商品占比太低

优先调整：

1. 调整 `target_consistency.py` 的 `is_low_target_occupancy`。
2. 建议先把 `target_primary_largest < 0.10` 提高到 `0.12` 或 `0.15`。
3. 检查 `model_scores.csv` 里的 `target_primary_largest_ratio` 和 `target_secondary_component_ratio`。

风险：对包装主色面积较小的商品可能更容易误排。

### 13.3 最终结果包含多个商品

优先调整：

1. 降低 `has_multiple_target_instances` 的第二主色组件阈值 `0.075`。
2. 降低 `has_multiple_products` 的 `second_component_ratio > 0.10`。
3. 检查 `standard` 图是否能代表目标商品颜色。

风险：复杂背景、手部、桌面物体可能被误判为多商品。

### 13.4 商品和背景边界不清

优先调整：

1. 保持或提高 `foreground_contrast < 55`。
2. 若要更严格，可提高到 `60` 或 `65`。

风险：透明、白色、浅色包装商品更容易被误排。

### 13.5 包含手机仍漏检

优先调整：

1. 降低手机面积占比 `0.055`。
2. 降低外接框占比 `0.14`。
3. 保持伸长比 `>2.5`，避免误杀黑色包装。

风险：黑色长条包装、暗色背景可能误判为手机。

### 13.6 可用图片不足 40 张

不要通过放宽所有阈值硬凑。建议：

1. 优先确认下载是否完整。
2. 查看 `model_scores.csv` 中最多的 `hard_flags`。
3. 只针对主要误排原因微调一个阈值。
4. 保留 `shortfall`，不要引入明显重复、模糊、不完整图片。

## 14. 建议的调整流程

每次只调整一个方向：

1. 选 1 到 3 个有问题的商品作为样本。
2. 记录当前 `selection_report.json`、`verification_report.json`、`model_scores.csv`。
3. 修改一个阈值或一个硬排除项。
4. 重新跑这些商品。
5. 对比最终结果、排除原因和 `shortfall`。
6. 确认没有明显副作用后，再扩大到 10 个、100 个商品。

推荐优先查看这些字段：

- `model_score`
- `hard_flags`
- `soft_flags`
- `is_blurry`
- `is_low_file_size`
- `is_low_boundary_contrast`
- `is_incomplete_product`
- `has_multiple_products`
- `has_other_product_color`
- `has_multiple_target_instances`
- `is_low_target_occupancy`
- `target_primary_largest_ratio`
- `target_secondary_component_ratio`
- `phone_component_ratio`
